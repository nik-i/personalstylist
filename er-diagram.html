Entity notes

EntityPurpose & key decisions
USERCore account only. Location is read live from the phone at recommendation time — nothing stored.
USER_PROFILEStyle signals extracted from best-outfit photos — coloring, body shape, highlight/downplay prefs.
SIZEOne row per wear category (tops, bottoms, dresses, shoes) per user. Region-aware for international sizing.
BRANDBrands as their own entity — reusable across users. No affiliate data at this stage.
BRAND_PREFERENCEJunction: USER ↔ BRAND with wear_category (day-to-day, office, occasion, athletic, activewear). A user can have multiple brands per category.
BEST_OUTFIT_PHOTOPhotos uploaded during onboarding. AI extracts style signals stored in USER_PROFILE.
WARDROBE_ITEMStructured columns for stylist-critical attributes (formality_level, season, pattern, fabric_type, warmth_level) + flexible JSON tags for everything else. warmth_level (light / medium / warm / heavy) is what the AI uses to match items to live weather — more precise than season alone.
OCCASIONWhat the user is dressing for — occasion_type, description, when (event_date), formality_level. No location stored; current location fetched from phone at recommendation time.
OUTFIT_SUGGESTIONA complete outfit built from owned wardrobe items for the occasion. Versioned — each change request creates a new version. weather_snapshot stores temp + conditions at generation time (e.g. 14°C, rainy) so feedback is contextually meaningful.
OUTFIT_ITEMJunction: which wardrobe items make up a given outfit suggestion.
OUTFIT_FEEDBACKCaptures feedback_source (audio | text), transcript only — audio is transcribed in real-time and never stored. will_wear is the strongest positive signal and feeds future outfit suggestions.
